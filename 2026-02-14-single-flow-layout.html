Sit App – Single Flow Layout

Simplified variant: one configurable flow per user. You can explore and adopt others' flows, but you only ever have one active flow. Modifying an adopted flow auto-flips visibility to private.

Key difference from multi-flow layout: No "My Flows" list. Settings → "My Flow" goes directly to the flow editor. Notifications use a single stepper instead of per-flow controls. Explore "Use" replaces your current flow (destructive, with confirmation).

Architecture Notes

Single Flow Model: Each user has exactly one flow (stored as JSON on the server). The "Explore → Use" action replaces the user's flow entirely. The previous flow is not preserved — it's overwritten. Attribution metadata (original author + flow name) is stored alongside the flow JSON.
Auto-Save: Flow edits are saved automatically as the user makes changes (debounced). The "Done" button simply navigates back to Settings. No explicit save action needed.
Visibility Auto-Flip: When a user modifies an adopted flow (one with attribution), visibility automatically changes to "Private". The user can manually set it back to "Public" if they want to share their modified version.
AI SQL Tool: The chat backend uses an LLM with a tool/function that can run read-only SQL queries against the prompt_responses table. The LLM decides when to query, formulates SQL, and interprets results in natural language.
Auth: Username/password stored server-side (hashed). Token stored in iOS Keychain, shared with Watch via WatchConnectivity or shared Keychain group. All API calls get an Authorization header.
Timers: Pure client-side on Watch. Uses WKExtendedRuntimeSession or background refresh for haptic vibration when timer ends. No server component.

UX Behavior Notes

Keyboard Behavior
When the chat input is tapped, the keyboard slides up and the message list scrolls to keep the latest message visible. The input bar stays pinned directly above the keyboard. Preset question chips are hidden when the keyboard is open, replaced by the scrolled message history. Dismissing the keyboard (tap outside or swipe down) restores the full layout.
Watch Notification Sync
Notification schedules are generated on the iPhone and synced to the Watch via WatchConnectivity (transferUserInfo or applicationContext). When the user changes notification settings on iPhone, the new schedule is pushed to the paired Watch. The Watch uses this schedule to fire local notifications independently, even when the iPhone is out of range.
Flow Replacement (Explore → Use)
When the user taps "Use" on a public flow in Explore, a confirmation dialog appears: "Replace your current flow? Your edits to '[current flow name]' will be lost." On confirm, the server replaces the user's flow JSON with a copy of the selected public flow, plus attribution metadata. The old flow is gone — no undo. A success toast confirms: "Now using '[flow name]'!"
Step Deletion Behavior
When a step is deleted, remaining steps are renumbered sequentially. Any answers in other steps that pointed to the deleted step become dangling references — highlighted with a warning badge and defaulting to "Submit" until re-assigned. A confirmation dialog warns: "Step 3 is referenced by 2 answers in other steps. Delete anyway?"
